## 2. 已确认的硬约束

1. 前端 UI 的功能、主要布局、用户操作路径、字段语义和既有显示结果必须保持兼容。
2. 测试用例 8.1、8.2、8.5、8.6、8.7 已能连接真实仪表完成全流程，重构不得改变其有效 SCPI 命令、测试点、执行顺序、判定规则、结果字段和输出文件语义。
3. 测试用例 8.3、8.4、8.8、8.9、8.10、8.11 暂时保持当前行为，不在本轮扩展业务能力。
4. 多人协作以测试用例为主要分工边界，一个用例的私有逻辑不得继续写入共享主窗口文件。
5. 允许优化 UI，但字体和当前视觉习惯以保持为主，只处理明确的可用性、边界和生命周期问题。
6. 允许引入异步执行、状态机、工作线程、Qt Test 和持续集成检查。
7. 真实仪表地址：
   - 4071E 频谱分析仪：`192.168.110.32:5025`
   - 1466G-V 信号发生器：`192.168.110.53:5025`
8. 所有真实仪表验证必须遵守安全门禁：默认不启用 RF 输出；测试结束、取消、异常、断连和程序退出都必须执行安全关断。
9. 安全清理完成前不得发送最终 `finished()` 通知，也不得允许启动下一次测试。
10. 软件无法在网络连接已经失效时绝对保证仪表 RF 状态；此时必须产生严重告警，并要求操作人员人工确认仪表输出状态。
11. 先分离代码归属，再改变执行语义。禁止从当前 `MainWindow` 直接一次性迁移到完整异步架构。
12. 8.1 作为第一条可编译、可测试、可回滚的纵向样板，线程和状态机必须先在 8.1 验证，再复制到其他用例。

## 4. 已发现的问题

### 4.1 合并冲突热点

- 所有用例集中在 `mainwindow.cpp`，多人按用例开发时仍会修改同一个文件。
- `applyCaseTemplate()`、`applyCaseResultTemplate()` 和 `onTestCaseClicked()` 使用大型条件分支集中管理所有用例。
- 8.5、8.6、8.7 的私有数据结构、图表、表格和仪表流程混入共享主窗口。
- 公共辅助函数和用例私有辅助函数没有目录或类型边界。
- `onTestCaseClicked()` 中存在提前 `return` 后的大段不可达旧逻辑，增加误改风险。

### 4.2 生命周期问题

- 测试流程主要在 GUI 主线程执行。
- 多处使用 `QThread::msleep()`、`waitForReadyRead()` 和局部 `QEventLoop`。
- `testRunning` 只是布尔标志，无法中断正在等待的 Socket、局部事件循环或休眠。
- 暂停按钮当前只更新状态栏，不暂停实际流程。
- 停止按钮只能在 GUI 事件被处理后生效，长流程中可能无法及时响应。
- 用例切换时没有统一的退出、取消、资源释放和结果冻结协议。
- 程序退出、设备断连、超时和中途异常没有统一汇入一个清理路径。

### 4.3 设备通信问题

- Socket 所有权、命令发送、查询等待、二进制块读取、错误队列和业务命令都在 `MainWindow`。
- 同步查询通过共享布尔状态避免 `readyRead` 冲突，扩展后容易出现响应归属错误。
- 没有独立命令队列和单请求上下文。
- 没有统一的取消令牌、请求状态和超时结果类型。
- 1466 和 4071 的设备能力被表达为散落的函数，而不是独立驱动接口。
- 模拟流程与真实仪表流程混杂，不利于离线自动测试。

### 4.4 UI 与组件边界问题

- `MainWindow::enhanceUi()` 在 `.ui` 之外动态重组大量界面，界面来源分散。
- 参数页、运行页、结果页和日志页没有独立 Widget 类型。
- 图表容器由不同用例反复清空和重建，直接 `delete` 与 `deleteLater()` 混用。
- 页面通过 `findChild()` 和对象名查找动态控件，编译期无法检查依赖。
- 用例结果直接存入 `QTableWidgetItem`，UI 控件同时充当数据模型。
- 全局强制 `16pt` 字号叠加高密度表格和最小窗口尺寸，在小屏幕或缩放环境下可能截断。
- 部分操作显示为可用，但功能仍是占位，例如报告、历史、普通截图和连接检测。

### 4.5 可测试性问题

- 业务判定依赖 UI 控件内容，无法脱离窗口测试。
- SCPI 解析函数、测试点生成和结果判定没有自动化测试。
- 没有可替换的仪表传输层，必须连接真实设备才能覆盖大部分流程。
- 没有固定构建脚本，系统 PATH 可能误选 Anaconda Qt 5.9.7。
- 当前没有 CI，也没有最低 Qt 版本检查。

## 6. 目标架构

目标结构采用以下依赖方向：

`UI Shell -> Use Case Controller -> Test Case -> Instrument Service -> Transport`

反向依赖禁止：

- 设备层不得引用任何 QWidget。
- 测试用例不得直接访问 `Ui::MainWindow`。
- UI 页面不得直接发送 SCPI。
- 一个测试用例不得访问另一个测试用例的私有 Widget 或私有状态。
- 公共模块不得包含只服务单一用例的字段和分支。

`MainWindow` 只负责：

- 创建和组合页面。
- 响应顶层导航与工具栏。
- 将用户操作转发给应用控制器。
- 展示全局连接状态和当前用例状态。

- SCPI 命令。
- 测试点循环。
- 用例判定。
- 用例图表生成。
- CSV 序列化。
- 设备安全关断细节。
- 用例私有表格列和字段。

### 6.4 最小展示接口

完整 UI 页面拆分安排在后期，但从第一个用例拆分开始就建立最小展示边界：

- `ITestCaseView`：接收当前用例信息、参数状态、运行状态和按钮状态。
- `ITestEventSink`：接收日志、进度、表格行、截图和图表数据事件。
- 用例私有结果 Presenter：把该用例的结构化结果转换为当前 UI 所需的表头、单元格和统计文本，例如 `case81/ICase81ResultPresenter`。

公共展示接口不得引用 `Case8xResult`、不得出现用例编号方法。第一版可以由
`MainWindow` 或薄适配器实现这些接口，但用例代码不得直接包含
`Ui::MainWindow`、不得直接访问 QWidget，也不得使用 `findChild()` 获取主窗口
控件。后续拆分四个页面时只替换展示实现，不再次修改用例业务接口。

## 7. 测试用例接口边界

### 7.1 公共接口

每个用例实现统一接口，建议包含：

```cpp
enum class TestCaseId {
    Case81,
    Case82,
    Case83,
    Case84,
    Case85,
    Case86,
    Case87,
    Case88,
    Case89,
    Case810,
    Case811
};

class ITestCase : public QObject {
    Q_OBJECT
public:
    virtual ~ITestCase() = default;
    virtual TestCaseId id() const = 0;
    virtual QString displayName() const = 0;
    virtual TestCaseCapabilities capabilities() const = 0;
    virtual bool canStart() const = 0;
    virtual ValidationResult validate(const TestParameters &parameters) const = 0;
    virtual void start(const TestCaseContext &context) = 0;
    virtual void requestPause() = 0;
    virtual void requestResume() = 0;
    virtual void requestStop() = 0;
    virtual void submitUserInput(const UserInput &input) = 0;

signals:
    void stateChanged(TestState state);
    void progressChanged(int value, const QString &text);
    void eventProduced(const TestEvent &event);
    void resultUpdated(const TestResult &result);
    void finished(const TestCompletion &completion);
};
```

实际接口会按 Qt 5.12.9 和线程模型调整，但必须满足：

- UI 只接收结构化事件。
- 用例不直接操作主窗口控件。
- 停止请求具备统一入口。
- `start()` 只允许在 `canStart()` 为 true 且状态为 `Idle` 时调用；同一轮运行不可重复调用。
- `requestStop()` 必须幂等，重复调用安全，不重复清理、不重复发出完成通知。
- `requestPause()`、`requestResume()` 对不支持或不满足状态的调用必须保持无副作用，并返回或发出明确拒绝结果。
- `TestCaseContext` 只保存具备明确所有权的智能指针、值对象或受控服务句柄，不保存调用栈临时对象引用。
- 每轮运行只能发出一次最终 `finished()`，且必须发生在安全清理完成之后。
- `submitUserInput()` 必须校验运行轮次、输入类型和当前等待点，拒绝过期输入。
- 用例对象由 `TestCaseRegistry` 或应用控制器创建和销毁；工作线程归属由测试运行器统一管理。
- 禁止在对象仍有排队信号时直接跨线程删除，销毁必须回到对象所属线程。
- 所有 queued signal/slot 使用的自定义值类型必须声明并注册为 Qt MetaType。
- 用例 ID 使用 `TestCaseId` 或等价值类型，显示文本只用于 UI，不参与业务路由。
- 每个用例拥有自己的私有模型和测试点定义。

### 7.2 结果概念分离

以下概念必须使用不同类型表达：

- 运行状态 `TestState`：`Idle`、`Running`、`WaitingForUserInput`、`CleaningUp` 等。
- 运行完成原因 `CompletionReason`：`Completed`、`ExecutionFailed`、`Cancelled`。
- 测试执行结果 `TestVerdict`：`Pass`、`Fail`、`Inconclusive`、`NotRun`。

业务判定为 FAIL 不等于程序执行失败。例如测试完整执行但测量值不达标时：

- `CompletionReason::Completed`
- `TestVerdict::Fail`

只有通信异常、协议错误、必要响应无法解析等导致流程未按定义完成时，才使用 `CompletionReason::ExecutionFailed`。

### 7.3 共享与私有模块判定

只有同时满足以下条件才进入共享模块：

1. 至少两个已实现用例实际复用。
2. 语义一致，而不是代码看起来相似。
3. 不包含用例编号、专用表格列或专用判定阈值。
4. 可通过独立单元测试验证。

否则保留在对应 `case8x` 目录，避免建立过早、错误的抽象。

## 8. 测试生命周期设计

### 8.1 状态机

统一状态建议为：

```text
Idle
  -> Validating
  -> Preparing
  -> Running
  -> WaitingForUserInput
       -> Running
  -> Pausing
  -> Paused
  -> Stopping
  -> CleaningUp
  -> Completed | ExecutionFailed | Cancelled
  -> Idle
```

约束：

- 任意终止路径都必须进入 `CleaningUp`。
- `CleaningUp` 必须幂等，多次调用不能重复破坏状态。
- 安全清理完成后才能进入最终完成状态并发出 `finished()`。
- 最终完成状态用于保留本轮结果，不自动立即跳回 `Idle`。
- 用户复位、确认结果或启动下一次任务时，控制器才将状态从最终完成状态转换为 `Idle`。
- 新用例启动前必须确认上一用例已完成清理，并已由控制器复位到 `Idle`。
- 用例切换时若当前任务未结束，应先请求停止并等待清理完成。
- 关闭窗口时执行同一停止和清理流程。
- `WaitingForUserInput` 不占用工作线程等待，也不运行局部 `QEventLoop`。
- `WaitingForUserInput` 期间停止请求必须立即进入 `Stopping`。

### 8.2 停止语义

“停止”不是设置一个布尔变量，而是：

1. 设置取消令牌。
2. 停止提交后续测试点。
3. 取消或超时当前 SCPI 请求。
4. 唤醒等待中的流程。
5. 4071 停止或退出当前测量。
6. 1466 执行 `ABORt`，关闭通道 RF，并执行 `OUTPut:ALL OFF`。
7. 冻结已经完成的结果。
8. 将未执行点标记为取消或未执行。
9. 完成安全清理并记录清理结果。
10. 进入 `Cancelled` 或 `ExecutionFailed`。
11. 发出唯一一次最终 `finished()`。

### 8.3 暂停语义

- 在安全检查点暂停，不在一条 SCPI 命令传输中间强行挂起线程。
- 暂停后不再进入下一个测试点。
- 需要持续 RF 输出的用例由用例策略决定暂停时是否关闭 RF。
- UI 明确显示“暂停请求中”和“已暂停”两种状态。
- 恢复后从当前安全检查点继续。

### 8.4 线程模型

目标方案：

- GUI 对象全部留在主线程。
- 仪表会话和测试执行器运行在专用工作线程。
- 使用 queued signal/slot 跨线程传输不可变值对象。
- 不允许工作线程直接访问 QWidget。
- 不使用 `QThread::terminate()`。
- 不在 GUI 主线程使用 `msleep()`、`waitForReadyRead()` 或测试级局部事件循环。

设备 Socket 与对应会话对象必须在同一线程创建和销毁，避免 Qt Socket 线程归属错误。

线程模型不在所有用例拆分前一次性切换。实施时分两步：

1. 先通过接口适配层把通信和用例代码移出 `MainWindow`，内部仍可同步执行。
2. 8.1 同步拆分验证通过后，只对 8.1 引入工作线程和状态机，形成纵向样板。

8.1 样板稳定后，8.2 及后续用例才按同一模式异步化。

### 8.5 人工输入

8.6、8.7 的 BLER 输入采用事件恢复模型：

1. 用例执行到人工确认点，保存当前运行轮次、测试点 ID 和待输入字段。
2. 用例发出结构化 `UserInputRequested` 事件。
3. 状态进入 `WaitingForUserInput`，工作线程返回事件循环，不阻塞等待。
4. UI 提交带运行轮次和测试点 ID 的 `UserInput`。
5. 用例校验输入，状态回到 `Running` 并继续。
6. 过期、重复或对应错误测试点的输入必须拒绝并记录日志。
7. 等待期间请求停止应立即进入停止和清理流程。

### 8.6 安全清理三级策略

软件应尽最大努力关闭输出，但必须承认网络失效时无法绝对保证仪表物理状态：

1. 正常清理：通过正常命令队列发送 4071 停止测量、1466 `ABORt`、通道 RF OFF 和 `OUTPut:ALL OFF`，并查询状态确认。
2. 强制清理：正常队列超时后取消或中止当前普通请求，清空待执行普通命令，直接发送安全命令并使用独立短超时确认。
3. 失联处置：安全命令仍失败时断开连接，记录 `ERROR`/严重安全告警，在 UI 显示“无法确认 RF 已关闭”，阻止自动开始下一用例，并要求操作人员现场确认仪表输出状态后手工复位。

不得向用户宣称网络断开后软件已经保证 RF 关闭。

## 9. 设备层设计

### 9.1 SCPI Transport

负责：

- TCP 连接与断开。
- 文本命令写入。
- 行响应和 IEEE 二进制块读取。
- 单请求超时。
- 请求取消。
- 原始通信日志。

不负责：

- 4071/1466 命令语义。
- 测试用例判定。
- UI 日志格式。

第一版 `IScpiTransport` 可以封装现有同步 Socket 调用，只要求移除对 `MainWindow` 的依赖并具备可替换接口。完整异步读取、请求队列和取消能力不在第一版强制实现。

### 9.2 Instrument Session

第一版 `InstrumentSession` 负责：

- 对现有同步发送、查询和二进制块读取提供稳定接口。
- 隐藏 Socket 与 `MainWindow` 的关系。
- 提供统一超时和结果类型。
- 允许注入 Fake Transport。
- 保持当前命令执行顺序和响应解析方式。

8.1 工作线程样板验证后，再演进为异步 Session，增加：

- 串行命令队列。
- 每条请求的唯一上下文。
- 防止多个查询响应交叉。
- 超时、断连、取消和协议错误统一结果。
- 错误队列读取。
- 测试结束时拒绝新的普通请求，只允许安全清理请求。

同步适配层与异步 Session 使用相同的上层 Driver 接口，避免用例再次重写。

同步到异步的演进契约：

- `Analyzer4071`、`Generator1466` 和测试用例只依赖稳定的 `InstrumentSession` 抽象，不依赖 `QTcpSocket`、同步等待或具体队列实现。
- 请求和响应从第一版开始就使用稳定值类型，例如 `ScpiRequest`、`ScpiReply`、`InstrumentError` 和 `BinaryBlockReply`。
- 同步实现与异步实现必须保持相同的命令语义、超时语义、错误分类和 Driver 方法签名。
- 第一版同步实现允许调用方在当前线程等待，但等待行为封装在 Session 内部，不泄漏到 Driver 和用例接口。
- 异步化时只替换 Session 内部调度、读取和取消实现；Driver、用例、结果模型和展示接口原则上不改签名。
- 若需要兼容过渡，可由 Session 内部提供同步适配器包装异步请求，禁止在 Driver 中维护同步、异步两套命令实现。
- 安全清理接口从第一版起独立于普通业务请求，异步化后仅改变调度优先级和取消机制，不改变上层调用契约。
- 阶段 3 必须用编译期接口测试和 Fake Transport 证明同步实现可替换，阶段 7 才允许移除同步内部实现。

### 9.3 4071 Driver

从 `MainWindow` 迁移：

- 设备识别。
- 频谱配置。
- 单次扫描。
- Marker 峰值与频率读取。
- ACPower 配置、测量和响应解析。
- 截图保存、下载和远端删除。
- 测量停止与能力检查。

### 9.4 1466 Driver

从 `MainWindow` 迁移：

- 设备识别。
- CW 频率、功率和输出控制。
- ARB 波形选择、采样率和输出控制。
- 输出状态回读。
- 错误队列。
- `ABORt` 和 RF 安全关断。

### 9.5 Fake Transport

离线测试使用 `FakeScpiTransport`：

- 按命令返回脚本化响应。
- 支持延迟、超时、断连、错误响应和二进制块。
- 记录命令顺序。
- 可断言停止后一定出现 RF OFF 命令。
- 可复现 8.1、8.2、8.5、8.6、8.7 的关键路径。

SCPI 轨迹断言分为：

- 必须存在的关键命令。
- 必须满足的相对顺序。
- 允许次数波动的轮询、错误队列和状态查询。
- 严格禁止出现的危险命令。

除稳定协议片段外，不要求所有轨迹逐字、逐次数完全一致。

## 11. qmake 与 CMake 的区别

### 11.1 qmake

- Qt 5 时代的传统构建工具。
- `.pro` 文件简单，当前工程已经在使用。
- 对小型 Qt Widgets 工程上手快。
- Qt Creator 支持成熟。
- 复杂目标、跨平台依赖、自动测试和大型工程组织能力弱于现代 CMake。
- Qt 6 仍能使用但已不是官方主推方向。

### 11.2 CMake

- 通用、现代、生态更广的构建系统。
- 更适合多目标、库拆分、测试、安装、打包和 CI。
- Qt 6 官方主推。
- 配置更明确，但迁移会带来额外构建变化。

### 11.3 本项目决定

本轮继续使用 qmake，原因是：

- 用户已明确要求。
- 当前目标是降低业务和 UI 耦合，不应同时引入构建系统迁移风险。
- qmake 足以支持多个源目录、静态内部库和 Qt Test 测试目标。

如果未来迁移 Qt 6 或工程继续扩大，可单独评估 CMake，不与本轮重构绑定。

## 12. Qt Test 与 CI 说明

### 12.1 Qt Test

Qt Test 是 Qt 自带的 C++ 测试框架，可用于自动验证：

- SCPI 响应解析。
- 测试点生成。
- 用例参数校验。
- PASS/FAIL 判定。
- CSV 转义。
- 状态机转换。
- 停止、超时和安全关断。
- Fake Transport 下的命令顺序。

测试程序不会显示正式主窗口，也不需要每次连接真实仪表。

### 12.2 CI

CI 是持续集成检查。每次提交或合并前自动执行：

1. 使用固定 Qt 5.12.9 qmake 生成构建文件。
2. 全量编译。
3. 运行 Qt Test。
4. 可选执行 offscreen UI 冒烟测试。
5. 检查失败时阻止有问题的代码合并。

初期可以先提供本地脚本，之后再根据代码托管平台接入 GitLab CI、GitHub Actions 或 Jenkins。

## 15. 真实仪表验证门禁

真实仪表测试分三级：

### L1：只读连接

- Ping/Socket 可达性。
- `*IDN?`。
- `SYSTem:ERRor?`。
- 配置状态查询。
- 不启用 RF。

### L2：最小输出

- 必须由用户确认接线、衰减和允许功率。
- 使用安全最低输出功率。
- 启用时间最短。
- `finally`/cleanup 路径无条件关闭 RF。
- 验证 `OUTPut1:STATe?` 为 OFF。
- 若连接失效导致无法确认，则按失联处置显示严重告警，不得将本次验证记为安全关断成功。

### L3：完整用例

- 仅在对应阶段代码完成并通过 Fake Transport 后执行。
- 按 8.1、8.2、8.5、8.6、8.7 分阶段回归。
- 保存命令日志、结果 CSV、截图和异常记录。

任何网络异常、超时、解析失败或用户停止都立即进入安全清理。

## 17. 风险与控制

| 风险 | 控制措施 |
| --- | --- |
| 重构改变已验证 SCPI 顺序 | 阶段 0 记录关键命令和顺序约束，Fake Transport 忽略允许波动的轮询次数 |
| 一次同时改变归属、线程和协议读取 | 先同步适配和拆分 8.1，再只对 8.1 异步化 |
| 异步后出现线程归属错误 | Socket 和 Session 同线程创建销毁，UI 只收值对象 |
| 停止时 RF 未关闭 | 三级清理策略、清理先于 finished、自动测试和真实回读 |
| 网络失效无法确认 RF 状态 | 明确软件能力边界，阻止后续启动并要求人工确认 |
| 人工 BLER 输入阻塞线程 | `WaitingForUserInput` 事件恢复模型 |
| UI 拆分导致显示变化 | UI 拆分放在业务拆分之后，逐页截图和字段对比 |
| 用例公共抽象过度 | 默认私有，满足复用判定后才提升为公共模块 |
| 多人继续修改共享文件 | 用例注册机制和目录所有权约束 |
| PATH 误用 Qt 5.9.7 | 固定 qmake 路径和构建脚本 |
| 真实仪表验证影响现场 | 分级门禁，默认只读，输出前确认接线与功率 |
